# Sample Project on Signup — Product Brief

**Author:** PM Analysis | **Date:** 2026-03-10 | **Status:** Discovery
**Tagline:** 신규 유저가 가입 즉시 완성된 예시 프로젝트에서 에디터를 탐색할 수 있게 한다.

---

## 1. 문제 (Problem)

**어떤 문제를 해결하는가?**

현재 Narrex의 온보딩 플로우는 가입 직후 빈 대시보드("아직 프로젝트가 없습니다")를 보여주고, 유저가 직접 프로젝트를 생성해야 핵심 UI(타임라인, 캐릭터 맵, 에디터)를 경험할 수 있다. 문제는 프로젝트 생성 자체가 이미 인지적 부담이 크다는 점이다:

1. **아이디어 입력 장벽**: 유저가 "어떤 이야기를 써야 할지" 결정해야 한다. 이것은 Narrex의 가치를 검증하기 전에 발생하는 요구다.
2. **대기 시간**: 아이디어 입력 후 AI 구조화에 10-30초가 소요된다. 유저가 아직 제품의 가치를 확인하지 못한 상태에서 기다려야 한다.
3. **결과 해석 부담**: 자동 생성된 타임라인, 캐릭터 맵이 처음 보는 유저에게는 "이게 잘 된 건지" 판단할 기준이 없다.

결과: 신규 유저의 상당수가 핵심 가치(타임라인 위에서 시각적으로 이야기를 구조화하고, AI가 씬별 초고를 생성하는 경험)를 접하기 전에 이탈할 위험이 있다.

**누구의 문제인가?**

첫 방문 유저 전체. 특히 "Narrex가 뭔지 구경하러 온" 탐색형 유저(Google OAuth로 가입했지만 아직 쓸 이야기가 확정되지 않은 사람)에게 가장 치명적이다.

**현재는 어떻게 해결하고 있는가?**

해결하고 있지 않다. Phase 1 PRD에는 온보딩 튜토리얼(REQ-005)이 명시적으로 Phase 2로 연기되어 있고, 현재 유저가 제품을 이해하는 유일한 경로는 "직접 프로젝트를 만들어보는 것"이다.

**왜 지금인가?**

Phase 1 개발이 진행 중이며, 신규 유저 유입 시점(베타 런칭)에 첫인상이 제품 채택을 결정한다. 온보딩 최적화는 런칭 후가 아니라 런칭 전에 설계되어야 한다. Phase 1의 핵심 성공 지표인 "코어 루프 완료율 40%"(프로젝트 생성 → 구조화 → AI 초고 생성)를 달성하려면, 유저가 코어 루프에 진입하기 전의 마찰을 최소화해야 한다.

## 2. 아이디어 평가 (Evaluation)

### 찬성 (Pros)

| # | 장점 | 근거 |
|---|------|------|
| 1 | **Time-to-Value 단축** | 유저가 가입 후 0초 만에 완성된 워크스페이스를 볼 수 있다. "이 도구가 뭘 하는지"를 설명 대신 체험으로 전달한다. |
| 2 | **인지 부담 제거** | "어떤 이야기를 쓸까"를 결정하지 않아도 제품을 탐색할 수 있다. 탐색형 유저의 이탈을 방지한다. |
| 3 | **UI 학습 수단** | 타임라인, 캐릭터 맵, 에디터, 씬 디테일 패널 등 복잡한 UI를 실제 데이터가 채워진 상태로 만져볼 수 있다. 빈 UI보다 학습 효과가 훨씬 높다. |
| 4 | **REQ-005(온보딩 튜토리얼) 대체** | Phase 1에서 연기된 가이드 튜토리얼 없이도, 예시 프로젝트 자체가 인터랙티브 튜토리얼 역할을 한다. |
| 5 | **AI 초고 품질 데모** | 미리 생성된 씬 초고를 보여줌으로써, 유저가 "AI가 이 정도 수준을 쓴다"는 기대치를 형성할 수 있다. 이것이 곧 유료 전환 동기가 된다. |

### 반대 (Cons)

| # | 단점 | 심각도 | 완화 방안 |
|---|------|--------|-----------|
| 1 | **엔지니어링 비용** | 중 | 가입 시 서버에서 프로젝트+트랙+씬+캐릭터+관계선+초고 데이터를 원자적으로 삽입해야 한다. DB 트랜잭션은 이미 설계되어 있지만(Section 4.1), 시드 데이터 정의와 삽입 로직 추가 필요. |
| 2 | **샘플 콘텐츠 품질 관리** | 중 | 샘플 프로젝트의 이야기가 매력적이지 않으면 오히려 역효과. "이 도구가 만드는 결과물이 이 정도인가"라는 부정적 첫인상. |
| 3 | **삭제/혼동 리스크** | 하 | 유저가 샘플 프로젝트를 "내 프로젝트"로 착각하거나, 삭제 후 복구를 원할 수 있다. 명확한 라벨링("예시 프로젝트")으로 완화. |
| 4 | **AI 생성 비용 0** | - (장점) | 샘플은 사전 제작이므로 유저별 AI 호출이 발생하지 않는다. 쿼터에도 영향 없다. |
| 5 | **유지보수 부담** | 하 | 샘플 콘텐츠를 업데이트(장르 추가, 품질 개선)하려면 시드 데이터를 관리해야 한다. 하지만 빈도가 낮으므로 부담은 작다. |

### 종합 판단

**좋은 아이디어다.** Time-to-Value 단축은 Phase 1 성공 지표(코어 루프 완료율 40%)에 직접 기여하며, 엔지니어링 비용은 상대적으로 낮다. 단, 샘플 콘텐츠의 품질이 핵심이다 -- 이것은 제품의 첫인상이 된다.

## 3. 샘플 프로젝트 설계 (What Should It Contain?)

### 장르 선택: 회귀 판타지 (Regression Fantasy)

**이유:**
- 한국 웹소설 시장에서 가장 인기 있는 장르 중 하나 (타겟 유저의 공감대 극대화)
- 회귀물은 본질적으로 다중 타임라인 구조 -- Narrex의 NLE 타임라인 UI를 자연스럽게 보여줄 수 있다
- 명확한 주인공 목표, 갈등, 캐릭터 관계가 있어 캐릭터 맵이 의미 있게 채워진다
- PRD의 페르소나(지연)가 정확히 이 장르를 상정하고 있다

### 구성

| 요소 | 수량 | 근거 |
|------|------|------|
| **트랙** | 2개 | 주인공 트랙 + 적대 세력 트랙. 멀티 트랙의 가치를 보여주되, 압도적이지 않은 수준. |
| **씬** | 8-10개 | 주인공 트랙 6개 + 적대 세력 트랙 2-3개 + merge point 1개. 타임라인을 스크롤 없이 한 화면에 볼 수 있는 적정 수량. |
| **캐릭터** | 4-5명 | 주인공, 유년 시절 친구(후에 적대), 스승, 히로인, 적대 세력 수장. 관계선이 의미 있게 그려질 수 있는 최소 인원. |
| **관계선** | 5-6개 | 동맹, 라이벌, 사제, 우정 등 다양한 관계 유형을 보여준다. solid, dashed, arrowed 모두 포함. |
| **AI 초고가 있는 씬** | 2-3개 | 전체 씬 중 일부만 초고가 작성된 상태. "Edited" 2개, "AI Draft" 1개, 나머지 "Empty". 이것이 유저에게 "여기서부터 내가 이어갈 수 있다"는 신호를 준다. |
| **Config** | 완전히 채워짐 | 장르: 회귀 판타지, 분위기: 긴장감 있는 성장물, 시대: 중세 판타지, 시점: 3인칭 제한, 톤: 절제된 서사 |

### 씬 구성 예시

| # | 트랙 | 제목 | 상태 | 설명 |
|---|------|------|------|------|
| 1 | 주인공 | 회귀의 순간 | Edited | 전투에서 죽은 기사가 12살의 몸으로 깨어남. 충격과 깨달음. (초고 포함) |
| 2 | 주인공 | 첫 번째 선택 | AI Draft | 전생에서 놓쳤던 기회를 다시 마주함. (AI 초고 포함) |
| 3 | 주인공 | 스승과의 재회 | Empty | 전생에서 죽었던 스승을 다시 만남. |
| 4 | 적대 세력 | 그림자의 움직임 | Empty | 적대 세력이 이미 음모를 진행 중. (병렬 트랙) |
| 5 | 주인공 | 숨겨진 보물 | Empty | 미래의 기억으로 보물의 위치를 찾아감. |
| 6 | 적대 세력 | 첩자 배치 | Empty | 적대 세력이 주인공 주변에 첩자를 심음. |
| 7 | 주인공 | 동맹 결성 | Empty | 전생에서의 동료를 미리 모음. |
| 8 | (merge) | 첫 번째 대결 | Empty | 주인공과 적대 세력이 처음 충돌. (merge point) |

## 4. 템플릿 vs 완전 생성 (Template vs Fully Generated)

### 결론: **사전 제작 시드 데이터 (Pre-crafted Seed Data)**

| 접근법 | 설명 | 판정 |
|--------|------|------|
| **A. 유저별 AI 생성** | 가입 시 AI가 랜덤 이야기를 실시간 생성 | 부적합. AI 비용 발생, 10-30초 대기, 품질 불확실. 온보딩 마찰을 줄이려는 목적에 역행. |
| **B. 사전 제작 시드 데이터** | 개발 시점에 수작업으로 만든 고품질 샘플을 모든 유저에게 동일하게 삽입 | **채택.** 비용 0, 대기 0, 품질 통제 가능. |
| **C. 장르별 복수 템플릿** | 여러 장르의 템플릿 중 선택 | Phase 2로 연기. Phase 1에서는 하나의 고품질 샘플이면 충분. 선택지 자체가 의사결정 부담. |

**이유:**
- 샘플 프로젝트의 목적은 "제품 체험"이지 "유저의 이야기 시작"이 아니다
- 모든 유저에게 동일한 콘텐츠를 보여주면 품질을 완벽하게 통제할 수 있다
- 유저가 샘플을 수정해도 되고, 삭제하고 자기 프로젝트를 만들어도 된다
- AI 호출이 없으므로 쿼터에 영향 없고, 비용 0

## 5. 기존 온보딩 플로우 통합 (Integration)

### 제안 플로우

```
Google OAuth 가입
    |
    v
[서버] 샘플 프로젝트 자동 생성 (DB seed insert)
    |
    v
대시보드 (프로젝트 목록)
    |
    +-- "회귀 기사의 두 번째 인생" [예시] <-- 샘플 프로젝트 카드
    |       (라벨: "예시 프로젝트" 뱃지)
    |
    +-- [+ 새 프로젝트]
    |
    v
유저 선택:
  A) 샘플 프로젝트 클릭 → 워크스페이스 진입 → 탐색/수정
  B) [+ 새 프로젝트] 클릭 → 기존 프로젝트 생성 플로우
```

### 핵심 설계 결정

1. **생성 시점**: Google OAuth 가입 완료 직후, 첫 번째 access token 발급과 함께 서버에서 자동 생성. 유저가 대시보드에 도달했을 때 이미 프로젝트가 존재해야 한다.

2. **라벨링**: 프로젝트 카드에 "예시" 뱃지를 표시하여 유저 본인의 프로젝트와 구분. `project` 테이블에 `is_sample BOOLEAN DEFAULT false` 컬럼 추가 또는 `source_type`에 `'sample'` 값 추가.

3. **수정 가능**: 샘플 프로젝트는 일반 프로젝트와 동일하게 편집/삭제 가능. "읽기 전용 데모"가 아니라 "편집 가능한 예시". 유저가 씬을 추가하고, 캐릭터를 수정하고, AI 초고를 생성할 수 있다.

4. **삭제 가능**: 일반 프로젝트와 동일하게 삭제 가능. 삭제 후 복구 불가(soft delete 30일은 동일하게 적용). 삭제해도 다시 생성되지 않는다.

5. **AI 쿼터**: 샘플 프로젝트에 포함된 사전 생성 초고는 유저의 AI 쿼터에 영향 없다. 하지만 유저가 샘플 프로젝트 내에서 새로운 AI 생성을 요청하면 정상적으로 쿼터가 차감된다.

6. **대시보드 Empty State 변경**: 기존 "아직 프로젝트가 없습니다" 빈 상태가 사실상 사라진다 (첫 방문 시 항상 샘플 프로젝트가 있으므로). 단, 유저가 샘플을 삭제한 후에는 빈 상태가 표시될 수 있다.

## 6. 대안 접근법 (Alternatives)

| 접근법 | 설명 | 장점 | 단점 | 판정 |
|--------|------|------|------|------|
| **A. 샘플 프로젝트 자동 생성 (제안)** | 가입 시 사전 제작 프로젝트 삽입 | 즉시 체험, 비용 0, 품질 통제 | 엔지니어링 약간, 콘텐츠 제작 | 채택 |
| **B. 인터랙티브 온보딩 투어** | Shepherd.js 같은 라이브러리로 단계별 UI 가이드 | UI 요소 직접 설명 가능 | 빈 UI 위에서 투어는 추상적, 실제 데이터 없이 가치 전달 어려움 | A와 결합 시 Phase 2 후보 |
| **C. 짧은 동영상/GIF 데모** | 대시보드에 제품 소개 영상 | 제작 간단 | 수동적 시청, 직접 만져보는 것보다 효과 낮음 | 보조 수단으로만 |
| **D. 간소화된 퀵스타트** | "한 줄 입력으로 3분 만에 시작" 같은 초간단 프로젝트 생성 | 유저 의도 반영 | 여전히 의사결정 필요, AI 비용 발생 | 기존 플로우와 중복 |
| **E. 갤러리/쇼케이스** | 다른 유저의 공개 프로젝트를 읽기 전용으로 탐색 | 다양한 예시 | 읽기 전용이라 편집 경험 불가, 소셜 기능 필요 | Phase 3+ |

**최적 조합**: A(샘플 프로젝트)를 Phase 1에 포함하고, B(인터랙티브 투어)를 Phase 2에서 샘플 프로젝트 위에 겹쳐서 구현. 샘플 프로젝트가 있으면 투어가 실제 데이터를 가리키며 설명할 수 있어 시너지가 크다.

## 7. 성공 신호 (Success Signal)

샘플 프로젝트가 효과적이라면: 신규 유저의 첫 세션에서 워크스페이스 진입률(대시보드 → 워크스페이스)이 크게 높아지고, 유저가 타임라인/캐릭터 맵/에디터를 실제로 클릭하는 비율이 증가한다. 궁극적으로 이것이 코어 루프 완료율(프로젝트 생성 → AI 초고 생성) 40% 달성에 기여해야 한다.

측정할 것:
- 샘플 프로젝트 열람률 (가입 유저 중 샘플 프로젝트를 여는 비율)
- 샘플 프로젝트 내 인터랙션 (씬 클릭, 캐릭터 클릭, 에디터 열기)
- 샘플 → 자체 프로젝트 전환율 (샘플 탐색 후 [+ 새 프로젝트]를 클릭하는 비율)
- 샘플 프로젝트 삭제 타이밍 (즉시 삭제 = 방해물, 탐색 후 삭제 = 역할 완수)
