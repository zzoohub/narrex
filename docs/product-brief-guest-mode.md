# Guest Browsing Mode -- Product Brief

**Author:** PM Analysis | **Date:** 2026-03-12 | **Status:** Approved
**Tagline:** 로그인 없이 Narrex를 체험하고, AI 생성 시점에만 인증을 요구한다.

---

## 1. 문제 (Problem)

**어떤 문제를 해결하는가?**

현재 Narrex는 모든 페이지에서 인증을 요구한다. 비로그인 사용자는 즉시 로그인 페이지로 리다이렉트되며, 제품이 어떤 도구인지 확인할 방법이 없다. 이것은 전환 퍼널의 최상단에서 가장 큰 마찰을 만든다:

1. **가치 확인 전 커밋 요구**: Google OAuth 동의는 심리적 비용이 있다. "이게 뭔지도 모르는데 구글 계정을 연결하라고?"
2. **탐색형 유저 전량 이탈**: 블로그, SNS, 입소문으로 유입된 잠재 유저가 랜딩 후 제품을 만져보지도 못하고 떠난다.
3. **바이럴 루프 차단**: 향후 프로젝트 공유 링크나 데모 URL을 만들어도, 수신자가 로그인벽에 막혀 콘텐츠를 볼 수 없다.

이 문제는 샘플 프로젝트(REQ-063)와 독립적이다. 샘플 프로젝트는 "가입 후" 첫 경험을 개선하지만, Guest Mode는 "가입 전" 진입 장벽을 제거한다. 두 기능은 보완 관계다.

**누구의 문제인가?**

Narrex를 처음 알게 된 잠재 사용자 전체. 특히 "(1) 누군가의 추천으로 링크를 클릭한 사람", "(2) 검색/SNS에서 유입된 탐색형 방문자", "(3) 경쟁 도구와 비교 중인 유저"가 핵심 대상이다.

**현재는 어떻게 해결하고 있는가?**

해결하고 있지 않다. 로그인벽이 100% 적용되어 있으며, 제품 스크린샷이나 소개 페이지도 별도로 없다.

**왜 지금인가?**

Phase 1 베타 런칭을 앞두고 있다. 첫 유저 유입 시점에서 전환율은 초기 성장의 핵심이다. 가입 전 제품 체험이 불가능하면, 마케팅 채널에서 유입된 트래픽의 전환 효율이 구조적으로 낮을 수밖에 없다. 런칭 후에 이 문제를 발견하면 이미 초기 트래픽을 낭비한 뒤다.

## 2. 제안 평가 (Evaluation)

### 2.1 전략적 판단: 좋은 방향인가?

**예, 올바른 방향이다.** "Try Before You Buy"는 SaaS의 검증된 전환 패턴이다. 특히 Narrex처럼 차별화된 UI(NLE 타임라인, 캐릭터 맵)가 핵심 가치인 제품에서는, 직접 만져보는 경험이 스크린샷이나 설명보다 압도적으로 효과적이다.

단, 제안의 세 가지 요소를 각각 평가해야 한다:

| 요소 | 판단 | 근거 |
|------|------|------|
| **Guest Browsing (비로그인 탐색)** | 강력 지지 | 전환 퍼널 최상단의 마찰을 제거한다. |
| **Deferred Auth (AI 호출 시점 인증)** | 지지 | 자연스러운 게이트. AI 생성은 서버 리소스를 사용하므로 인증이 합리적인 시점이다. |
| **Client-only Demo Project (클라이언트 메모리 전용)** | 지지, 단 범위 한정 필요 | DB 저장 없이 체험용 데이터를 클라이언트에서 관리하는 것은 올바른 트레이드오프다. |

### 2.2 Guest 접근 범위: 어디까지 허용할 것인가?

**권고: "View + Local Edit" -- 읽기 및 로컬 편집 허용, AI 호출만 게이트**

| 수준 | 설명 | 판단 |
|------|------|------|
| **View Only** | 데모 프로젝트를 보기만 가능. 클릭/편집 불가. | 부족. "구경"만으로는 제품의 인터랙티브 가치를 전달할 수 없다. |
| **View + Local Edit (권고)** | 데모 프로젝트에서 씬 드래그, 캐릭터 편집, 에디터 입력 등 UI 조작 가능. 변경사항은 메모리에만 존재. AI 생성/편집 시 로그인 요구. | 최적. 제품의 핵심 인터랙션을 체험하되, 비용이 발생하는 지점에서만 게이트. |
| **Full Edit + Multi-project** | Guest가 여러 프로젝트를 메모리에 생성 가능. | 과잉. 복잡도 대비 전환 기여가 낮다. 하나의 데모 프로젝트로 충분하다. |

**권고 범위 상세:**

- Guest가 할 수 있는 것: 데모 프로젝트 탐색, 타임라인 조작(씬 이동/리사이즈), 캐릭터 맵 조회 및 편집, 에디터에서 텍스트 직접 입력/수정, Config Bar 변경
- Guest가 할 수 없는 것: AI 씬 생성, AI 편집 요청, 새 프로젝트 생성, 데이터 서버 저장
- 게이트 시점: AI 호출 버튼 클릭 시 모달 표시 -> 로그인 유도

### 2.3 클라이언트 메모리 전용: 올바른 접근인가?

**예, Guest 체험 데이터에 한해서 클라이언트 전용이 맞다.**

| 대안 | 장점 | 단점 | 판단 |
|------|------|------|------|
| **A. DB에 익명 유저 레코드 생성** | 서버 로직 통일 | 가비지 데이터 대량 발생, 정리 로직 필요, user_account 없이 project FK 불가 | 부적합 |
| **B. localStorage/sessionStorage** | 브라우저 새로고침 시 유지 | 용량 제한(5-10MB), 직렬화 복잡, 탭 간 동기화 이슈 | 보조적 사용 가능 |
| **C. Solid signals/stores (메모리)** | 기존 아키텍처와 일관, 구현 단순, 성능 최적 | 새로고침/탭 닫기 시 소멸 | **권고** |
| **D. IndexedDB** | 대용량, 구조화 저장, 오프라인 지원 | 과잉 엔지니어링, 가비지 관리 필요 | Phase 1에서 불필요 |

**확정: C (인메모리). 새로고침 시 초기 상태로 리셋한다. IndexedDB/localStorage 영속은 사용하지 않는다.**

이것은 엔지니어링 편의가 아니라 제품 판단이다. 리셋이 영속보다 Guest 체험 UX와 전환율 모두에서 유리한 이유:

1. **일관된 첫인상 보장.** Guest가 언제, 몇 번째 방문이든 동일한 데모 상태를 본다. 영속 시에는 "이전에 뭘 수정했는지 기억 안 나는 반쯤 변형된 데모"를 마주하는 문제가 생긴다. 이것은 첫인상을 통제할 수 없게 만든다.
2. **"체험" 프레이밍 강화.** 리셋되는 것 자체가 "이건 체험판이고, 진짜는 가입 후에"라는 메시지를 자연스럽게 전달한다. 영속은 오히려 "여기서 작업할 수 있다"는 잘못된 기대를 만들어, 로그인 게이트에서의 실망을 키운다.
3. **전환 퍼널 측정 용이.** 모든 Guest가 동일한 초기 상태에서 시작하므로, 데모 탐색 -> AI 시도 -> 가입의 전환율을 깨끗하게 측정할 수 있다. 영속 시에는 재방문자의 상태가 제각각이어서 분석이 오염된다.
4. **엣지 케이스 제거.** 데모 fixture가 업데이트되면? 영속 시에는 구버전 데이터와 신버전 코드의 호환성 문제가 생긴다. 리셋이면 항상 최신 fixture를 로드한다. 가비지 데이터 정리도 불필요하다.
5. **실사용 시나리오에서 리셋의 비용이 낮다.** "실수로 새로고침해서 탐색 중이던 상태를 잃었다"는 이론적으로 가능하지만, Guest는 데모를 "구경"하는 중이지 30분간 편집하는 중이 아니다. 만약 그 정도로 몰입했다면 이미 가입할 준비가 된 유저다.

**핵심 구현 고려사항:**

- 데모 프로젝트 데이터는 하드코딩된 fixture로 정의 (기존 REQ-063 샘플 프로젝트와 동일한 콘텐츠 재사용 가능)
- API 호출 없이 클라이언트에서 직접 로드. SolidQuery 캐시가 아닌 별도 signal/store로 관리
- 로그인 후에는 클라이언트 데모 데이터를 버리고, 서버의 실제 데이터(REQ-063 샘플 프로젝트 포함)로 전환

### 2.4 "작업 저장" 프롬프트: 로그인 게이트 전에 필요한가?

**Phase 1에서는 불필요하다.**

- Guest의 로컬 편집은 "체험"이지 "작업"이 아니다. 유저가 데모 프로젝트에서 한 수정은 탐색 과정의 부산물이다.
- "로그인하면 지금 수정한 내용이 저장됩니다"라는 약속은 기술적으로 복잡하다: 클라이언트 상태를 서버로 마이그레이션하는 로직이 필요하고, 서버의 샘플 프로젝트와 충돌한다.
- 대신, 로그인 유도 모달에서 "가입하면 나만의 프로젝트를 만들 수 있습니다"로 전환 동기를 부여하는 것이 더 깔끔하다.

**향후 고려**: Phase 2 이후 Guest가 자체 아이디어를 입력하여 구조화하는 기능을 추가한다면 (Guest + REQ-001 free text input), 그때는 "작업 저장" 프롬프트가 필요해진다.

### 2.5 전환 퍼널 영향

**기대 효과:**

```
[현재]
방문 -> 로그인벽 -> (대량 이탈) -> 가입 -> 빈 대시보드 -> (추가 이탈) -> 프로젝트 생성

[Guest Mode + 샘플 프로젝트 적용 후]
방문 -> 대시보드(데모 프로젝트 표시) -> 워크스페이스 진입 -> 가치 확인 -> AI 생성 시도 -> 로그인 모달 -> 가입 -> 서버 샘플 프로젝트 + 자체 프로젝트 생성
```

핵심 변화: "가입"이 퍼널의 첫 단계에서 중간 단계로 이동한다. 유저가 가치를 확인한 후에 가입하므로 전환율이 높아지고, 가입 후 이탈률이 낮아진다.

### 2.6 엣지 케이스 및 리스크

| 리스크 | 심각도 | 완화 방안 |
|--------|--------|-----------|
| Guest가 30분간 데모를 편집한 후 로그인했는데 편집 내용이 사라짐 | 중 | 로그인 모달에 "데모 체험 내용은 저장되지 않습니다. 가입 후 나만의 프로젝트를 만드세요"를 명시. Guest 편집을 "체험"으로 포지셔닝. |
| API 엔드포인트가 Guest 요청을 잘못 처리 | 중 | Guest 모드에서는 API를 호출하지 않으므로 서버 변경 최소화. AI 엔드포인트의 기존 AuthUser 미들웨어가 그대로 게이트 역할. |
| Guest 데이터와 서버 데이터의 타입 불일치 | 중 | 데모 fixture 데이터의 타입을 서버 응답 타입과 동일하게 유지. 공유 TypeScript 타입 사용. |
| SEO/크롤러가 로그인벽 뒤의 콘텐츠를 인덱싱 못함 | 하 | Guest Mode로 주요 페이지가 접근 가능해지면 자연스럽게 해결. |
| 악의적 사용자가 Guest 모드를 악용 | 하 | Guest는 서버 리소스를 사용하지 않으므로 악용 벡터가 거의 없다. |

## 3. 제안 방향 (Proposed Direction)

**단일 데모 프로젝트 기반의 Guest Browsing + AI 시점 Deferred Auth. Phase 1 스코프에 포함.**

비로그인 사용자가 앱에 진입하면 대시보드부터 보여준다. 대시보드에는 데모 프로젝트가 표시되어 있고, 클릭하면 워크스페이스로 진입하여 타임라인/캐릭터 맵/에디터를 자유롭게 탐색하고 로컬에서 편집할 수 있다. 데모 데이터는 클라이언트 인메모리(하드코딩 fixture)로 관리하며, 새로고침 시 항상 초기 상태로 리셋된다. AI 생성 또는 AI 편집 버튼을 누를 때만 로그인 모달을 표시한다. 로그인 완료 후에는 클라이언트 데모 데이터를 버리고, 서버의 실제 데이터(REQ-063 샘플 프로젝트 포함)로 전환한다.

**핵심 인사이트**: Narrex의 차별화는 시각적 스토리 구조화 UI다. 이것은 AI 호출 없이도 체험 가능하다. AI 생성은 가치의 두 번째 단계이며, 자연스러운 인증 게이트 시점이다.

---

## 4. 성공 신호 (Success Signal)

Guest Mode가 효과적이라면: 비로그인 방문자의 상당 비율이 데모 프로젝트를 열어 UI를 조작하고, 그중 의미 있는 비율이 AI 생성을 시도하여 로그인 모달을 만나며, 모달을 만난 유저의 높은 비율이 실제로 가입을 완료한다. 현재 "방문 -> 로그인벽 -> 이탈" 패턴이 "방문 -> 탐색 -> 가치 확인 -> 가입"으로 전환되는 것이 핵심 지표다.

## 5. REQ-063(샘플 프로젝트)과의 관계

| 구분 | REQ-063 (샘플 프로젝트) | Guest Mode (이 제안) |
|------|------------------------|---------------------|
| 대상 | 가입 완료한 신규 유저 | 비로그인 방문자 |
| 데이터 위치 | 서버 DB | 클라이언트 메모리 |
| 콘텐츠 | 회귀 판타지 샘플 | 동일 콘텐츠 재사용 가능 |
| 목적 | 가입 후 Time-to-Value 단축 | 가입 전 진입 장벽 제거 |
| 관계 | 보완적 | 보완적 |

두 기능이 합쳐지면 전체 퍼널이 매끄러워진다: Guest Mode(가입 전 체험) -> 가입 -> REQ-063(가입 후 즉시 서버 데이터로 이어서 작업).
